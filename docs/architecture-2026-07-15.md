# 2026-07-15 破坏性简化重构

## 为什么重构

旧应用同时存在后端求解线程、执行锁、进度锁、取消事件，以及前端求解轮询、快速点击队列、批量进度轮询和自动执行定时器。`execute_burst()` 还会在持有执行锁时释放锁并递归进入 `execute_step()`。业务状态被多个拥有者写入，任何一个异常分支都可能留下“UI 认为可继续、后端认为忙中”的分裂状态。

重构目标不是增加另一层抽象，而是减少状态拥有者。

## 新的不变量

1. 同一时间最多一个后台作业。
2. 只有 `OperationCoordinator` 创建后台线程和拥有取消令牌。
3. 前端只有一个短轮询器；作业结束后轮询器消失。
4. 分析先产生完整 `AnalysisBundle`，再一次性更新应用状态。
5. GUI 与 CLI 只调用 `planner.solve_board()`，搜索策略不再分叉。
6. 自动执行在新关卡先运行一个受限的开局粗解：只批量点击普通羊的直接离场闭包，整批后统一核验。
7. 狼、炸弹羊和安全阻断局面不进入粗解快点；教程手势只作为画面提示，不参与执行控制。
8. 暂停在当前点击后生效；粗解批次也会在当前点击落下后停止下一次点击。
9. 单样本学习候选继续构成硬 `execution_blocker`。

## 状态机

```text
idle
  └─ capturing
      └─ analyzing
          ├─ done / blocked
          ├─ opening-coarse (仅新关卡普通羊直出)
          │   └─ executing → verifying → done / solving
          └─ solving
              ├─ done / preview
              └─ executing
                  └─ verifying
                      ├─ done
                      ├─ solving (下一步)
                      └─ pausing → cancelled

任意阶段异常 → error
```

阶段只是对同一作业的投影，不会创建子作业。执行后的重求在当前作业内同步完成，因此不会出现“执行线程等待求解线程、求解线程又被 busy gate 拒绝”的环路。

## 分析边界

`analysis_engine.analyze_image()` 输入截图、标定和历史观测，输出不可变的 `AnalysisBundle`：

- `grid`
- `sheep`
- `debug`
- `board_data`
- `layout`
- `report`

透视缩放统一使用 `board_grid.from_params()`。GUI 不再复制四角缩放算法。只有完整 bundle 返回后，`app.py` 才更新实时状态并发布 JSON/PNG 产物。

## 求解边界

`planner.solve_board()` 统一以下策略：

- 14 只及以下：最优 A*；
- 单调朝向大盘：确定性 EXIT 闭包，然后依次尝试窄束搜索、固定种子的多路宏动作重启和 weighted A*；
- 其他大盘：先保留快速 greedy 候选，但继续使用 weighted A* 与 beam 消耗剩余预算；
- 动态狼道：只发布已经证明安全的直出前缀，随后在线重扫。

返回值是 `PlanResult`，包含步骤、最终棋盘、完成状态、剩余数、算法类型、超时和搜索信息。GUI 负责把它转换为点击坐标与缓存记录，CLI 负责打印和渲染；二者不再选择不同算法。

`ElasticBudget` 把初始时限、单次续时和累计上限收敛成一个有界 deadline。当前预算耗尽且仍未得到完整解时，planner 在同一作业内自动续时并启动下一轮组合搜索；续时不会创建第二个后台作业，也不会绕过取消令牌。所有预算变化与策略开始/结束都作为结构化过程事件发布，前端只负责渲染时间线。

`solver_learning.py` 是 planner 外部的可选策略画像模块。它按棋盘尺寸、棋子规模、物种和规则特征聚类，增量累计各搜索策略的成功率、剩余数改善、耗时和扩展节点；冷启动保持手工调优顺序，样本足够后才调整策略顺序和时间配比。求解只读取内存快照，观察结果由 daemon 线程写入 `cache/solver_strategy_learning.json`。该模块不弹提示、不要求用户确认、不进入 `execution_blockers`，任何加载或写入失败都不得影响求解与执行。

`planner.coarse_exit_plan()` 是自动模式唯一的开局粗解入口。它不运行搜索，只在逐步缩小的棋盘上收集普通羊 `EXIT`，因此能在精解前立刻发布可执行前缀。后端按通道依赖重排并计算最短安全点击间隔，批次结束只捕获一次真实棋盘，再对缩小后的棋盘运行完整求解。

同一入口也支持有限暴露层。界面的“快速解法”会复用当前已识别棋盘，只在尚无棋盘时采集，只处理最多三层普通羊 `EXIT`，不会继续精解；右侧计划按离场层折叠显示。

## 前端边界

### 双模式产品边界

前端不再把移动端视作缩小后的桌面端，而是投影为两种能力集合：

- `operator` 桌面操作：窗口采集、分析、求解、快速解法、连续执行、校准和完整复核。
- `reference` 移动参考：相册上传、分析标注、识别报告、求解和可逆沙盘。

移动参考默认面向 393 × 852 的 iPhone 15 Pro 逻辑视口。上传图片直接构成画布，状态、标注开关、参考操作和结果入口以四周原子卡片叠加；完整识别报告与步骤列表只在底部抽屉中披露。

两种模式不仅由 CSS 区分。上传走 `workflow_start("upload")`，不会查找或采集桌面窗口；后端把来源记为 `reference` 并拒绝 `quick | step | auto`。移动端求解还会显式传入 `capture_if_missing=false`，因此缺少棋盘时返回上传提示，不会退回隐式桌面采集。

前端只做：

- 提交 `analyze | solve | quick | auto`；需要逐步核验的特殊棋子路径仍可使用 `step`；
- 轮询一个 `workflow_status`；
- 将返回状态渲染成截图覆盖层、安全报告和计划列表；
- 将后端过程事件渲染成算法时间线，但不在前端决定策略或续时；
- 提交明确的人工校准/复核操作。

求解结束后，前端必须把完整解、部分计划、超时和结构冲突显示为不同的持久状态；未获授权的计划直接锁住执行按钮，并给出“重新求解 / 复核棋盘”的下一步。任何带步骤的结果默认选中第 1 步进入可逆沙盘。

前端不做：求解阶段选择、点击排队、批次编排、自动恢复判断或执行安全授权。

运行设置采用双层持久化：输入变化立即写入页面本地存储，同时通过桥接接口原子更新 `runtime_settings.json`。启动时比较 `updated_at_ms` 并选择较新的副本，再自动补齐另一侧；数值范围由前后端分别校验，因此损坏或旧版本数据不会阻断应用启动。

## 删除的路径

- 未验证、可包含任意动作的多点击 burst 实现；
- 脱离安全报告和计划授权的快速点击旁路；
- 前端快速点击队列与多个 `setInterval` / `setTimeout` 状态机；
- 旧批次进度镜像；
- 裸像素 click-and-refresh 路径；
- GUI 与 CLI 重复的搜索策略。

`execute_burst()` 仅保留一个兼容壳，实际执行一条经过预检和回读的单步，不再包含自己的编排逻辑。
开局粗解快点由同一个 `OperationCoordinator` 作业内的受限直出阶段拥有，不恢复旧前端队列或独立状态机。

## 验证标准

- 并发提交第二个作业必须返回 busy，而不是创建第二条线程。
- pause 阶段必须显示“当前点击完成后暂停”，随后进入 cancelled。
- 小盘交互型棋子仍走最优 A*。
- 无标定是正常阻断结果，不是崩溃。
- 当前真实截图必须能完成识别、生成无冲突棋盘并求出计划。
- 全套 pytest、Python 编译和前端 JS 语法检查必须通过。
