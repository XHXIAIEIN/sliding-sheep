# Sheep Solver Optimization Roadmap

> 2026-07-15：本文记录的是重构前的问题清单。进程编排、统一求解入口、默认批量点击和前端多状态机已经由 [architecture-2026-07-15.md](architecture-2026-07-15.md) 的破坏性简化重构取代；其余识别研究项仍可作为历史参考。

更新日期：2026-07-11

这份文档整理了当前项目可继续优化的功能点，目标是给下一次会话直接作为执行清单使用。

## 审查范围

- 桌面应用：`app/index.html`、`app.py`
- 识别链路：`scripts/capture_window.py` -> `scripts/detect_occupancy.py` -> `board.json`
- 求解链路：`scripts/solve_board.py`、`scripts/solver.py`、`scripts/solver_search.py`
- 缓存与回放：`cache/levels/`
- 静态验证：Python AST / import smoke / `scripts/test_solver.py` / 前端 `node --check`
- 数据样本：当前缓存快照、反馈文件、诊断图

## 当前结论

项目已经不是“半成品”，而是一个可运行的桌面求解器：有截图、标定、识别、沙盒、求解、真实点击、缓存复用和点击后反馈校验。但它现在最需要的不是继续加功能，而是先把“错误场景下不会误点”和“缓存/求解结果不会污染后续执行”这两件事做扎实。

## 本轮完成状态（2026-07-11）

本轮已继续完成此前最主要的 P1/P2 深度项：

- [x] 多检测器融合：箭头、body/分水岭、牛体/牛脸/逐格检测全部并行，统一校准置信度并保留来源投票；删除“箭头候选超过阈值就完全切换”的硬分支。
- [x] 全局占格优化：使用二元 MILP 最大化全盘候选总分，每格至多被一个棋子占用；箭头与完整牛体作为强锚点，失败时有明确 greedy fallback。
- [x] 完整拒绝审计：`sheep_candidates.json` 同时保存 raw、fused、kept、dropped、融合投票、全局冲突对象、拒绝原因与 optimization loss。
- [x] 分项置信度与全盘健康度：occupancy / axis / facing / species / hazard 独立记录，场景报告输出统一 `health_score`。
- [x] 多帧识别：GUI 保留最近 4 帧兼容观测，以时序存在率和多数票稳定朝向、物种；切换窗口或重标定会清空历史。
- [x] 动态 hazard 在线模型 v1：危险格形成 `stable / emerging / fading / transient` 时间线，帧间不稳定会加入硬执行 blocker，点击后重识别继续更新。
- [x] 完整手工棋盘编辑器：支持新增、移动、删棋子、改朝向/物种、切换 hazard、还原识别、schema 校验、显式确认和保存带 provenance 的人工样本。
- [x] 缓存回归工具：`scripts/recognition_regression.py` 并行重跑缓存截图并输出逐样本占格/朝向/场景差异；本轮报告覆盖 81 份快照。

本轮真实截图结果为 132 个 raw 候选 -> 88 个融合候选 -> 39 个全局保留棋子，0 占格冲突、0 人工复核 blocker；覆盖图已人工核对。缓存回归平均占格召回 97.12%、精度 96.01%、共同占格朝向一致率 96.80%，4 个 popup 均未被提升为 gameplay。

仍属于后续增量研究、而非上述主项目缺口的内容：更多物种/特效真实样本、SCC/永久障碍结构剪枝、EXIT 宏步、学习摘要读回策略、缓存产品化和 UI 动画/筛选体验。

## P0：应优先处理

### 1. 场景有效性判断缺失

- 问题：当前流程会把非棋盘场景也尽量识别成棋盘，甚至可能生成可执行计划。
- 风险：转场、结算、弹窗、加载中、特效页都可能触发误识别和误点击。
- 建议：
  - 增加 `scene_state`：`gameplay | transition | victory | popup | loading | unknown`
  - 非 `gameplay` 状态禁止出 `board.json`、禁止求解、禁止执行
  - 把“无法确认是棋盘”作为正常结果，而不是硬凑一个局面

### 2. “执行一次”实际可能连点多次

- 问题：当前单次执行会进入 burst 流程，可能在验证前连续触发多次真实点击。
- 风险：一旦识别错一步，后面几步都会建立在错误状态上。
- 建议：
  - 默认行为改成“点 1 步 -> 重新截图 -> 重新识别 -> 校验 -> 再决定下一步”
  - 快速连点改成单独的高级模式，不应作为默认执行路径

### 3. 低置信度结果没有强制阻断执行

- 问题：审查模式、低置信度、危险区重叠、标定漂移等状态，没有统一变成“禁止自动执行”的硬门槛。
- 风险：用户以为系统已自检，实际上仍可能带病点击。
- 建议：
  - 增加统一 `execution_blockers`
  - 命中任一 blocker 时，只允许查看和人工复核，不允许真实点击

### 4. 障碍物与棋子重叠未被显式判错

- 问题：当前布局校验只关注羊和羊冲突，没有把 hazard-piece overlap 当成非法盘面。
- 风险：求解器可能在不合法局面上继续搜索。
- 建议：
  - 在布局构建和 `board.json` 校验阶段直接报错
  - 把这类样本标记为不可执行缓存

### 5. 缓存哈希忽略关键规则信息

- 问题：当前 state hash 会发生真实碰撞，至少已经观察到 hazard 被忽略导致的冲突。
- 风险：不同盘面复用同一套旧动作，属于最高风险缓存错误。
- 建议：
  - 把 hazards、物种配置、规则开关、求解器版本纳入 hash
  - 升级缓存 schema，旧缓存只读不复用

### 6. 更差的求解结果会覆盖更好的缓存结果

- 问题：未解出的 partial plan 可能覆盖之前已解出的完整解。
- 风险：缓存质量会退化，后续命中变差甚至变错。
- 建议：
  - 改成不可变 revision 记录
  - 单独维护“最佳方案选择规则”，优先完整解、其次更短计划、再看版本和时间

### 7. 目标窗口选择不稳

- 问题：窗口标题 substring 匹配可能选中求解器自己，因为标题里也带目标关键字。
- 风险：截图错窗口、点击错窗口。
- 建议：
  - 增加 PID / 进程名 / 窗口类过滤，排除当前应用
  - UI 中加入显式窗口选择器

### 8. 点击坐标协议过于原始

- 问题：后端接受前端直接传入的像素坐标，再叠加窗口偏移执行。
- 风险：窗口移动、DPI 变化、前端状态滞后时会点错位置。
- 建议：
  - 前端只传 `board_revision + move_id`
  - 后端基于当前标定和当前窗口状态重新计算点击坐标
  - 点击前校验 HWND、窗口尺寸、DPI、前台状态、棋盘多边形

### 9. 点击前缺少完整 preflight

- 问题：目前只对部分特殊规则做点击前检查，大多数普通点击在不可逆操作后才发现问题。
- 风险：纠错成本高，失败样本污染反馈数据。
- 建议：
  - 每次真实点击前重新确认“当前棋盘仍与计划起点一致”
  - 不一致时立即停机，不允许继续回放剩余计划

### 10. 缺少真正可中断的急停

- 问题：前端 stop 不能可靠打断后端点击、sleep 或求解过程。
- 风险：用户已经想停，但系统还会继续动作。
- 建议：
  - 增加后端 cancel token / interrupt flag
  - 所有点击和等待都要检查取消状态
  - 鼠标按下后要保证 finally 中释放

### 11. 前端 busy 状态和热键状态机不一致

- 问题：刷新和全局热键对 busy 的判断不统一，可能在旧计划上继续执行。
- 风险：用户界面显示“忙中”，热键却还能触发真实动作。
- 建议：
  - 建立统一执行状态机和 `board_revision` 锁
  - UI 按钮、热键、后台任务共用同一套 gating

### 12. 首次运行标定流程不通

- 问题：没有 `grid_params.json` 时应用会退出，但标定页本来需要应用先启动。
- 风险：新用户无法完整走通首次使用流程。
- 建议：
  - 允许无标定参数启动
  - 首次启动进入“未标定模式”，只开放截图和标定入口

### 13. 独立 `grid_tuner.html` 已过时

- 问题：页面内仍写死旧尺寸和旧行列数，导出的 JSON 也缺字段。
- 风险：标定工具和实际识别网格不一致。
- 建议：
  - 和当前网格配置统一为同一份数据源
  - 要么彻底修正，要么移除独立旧版入口

### 14. 标定参数缺少合法性验证

- 问题：四点顺序、凸性、面积、越界、分辨率漂移等没有硬校验。
- 风险：坏标定会在后续阶段放大成大量识别错误。
- 建议：
  - 加入完整参数校验
  - 分辨率或窗口尺寸变化后强制要求重新确认或重标

### 15. `board.json` 输入约束不完整

- 问题：物种、朝向、滑动模式、相邻关系、越界、重叠等缺少系统校验。
- 风险：识别 bug 和缓存污染会直接传进求解器。
- 建议：
  - 建立统一 schema 校验层
  - solver 前一律走 `validate_board()`

### 16. weighted A* / beam 边界条件有已知错误

- 问题：存在“剩余数已为 0，但返回 `solved:false`”的边界 bug。
- 风险：完整解会被误记成未解。
- 建议：
  - 修正 `scripts/solver_search.py` 对空剩余状态的判定
  - 给一跳清盘局面补回归测试

### 17. 启发式没有把 hazard 成本纳入评分

- 问题：合法移动生成知道 hazard，但评分函数基本忽略 hazard。
- 风险：搜索会优先走看似近、实际高风险或死路的分支。
- 建议：
  - 在 heuristic 中加入 hazard block / trap / dead-end 代价
  - 与真实合法性规则保持一致

### 18. 大盘 `axis_both` 兼容路径有崩溃风险

- 问题：大于 14 子时会走另一套搜索路径，`axis_both` 可能导致 `KeyError: None`。
- 风险：中大盘直接炸求解。
- 建议：
  - 修复 `solve_board.py` 的兼容分支
  - 给大盘样本加专项回归测试

## P1：第二阶段优化

### 识别与审查

- 用多检测器融合替代“候选数超过阈值就完全切换另一套规则”的硬分支
- 用全局最优化替代局部贪心占格
- 在 `sheep_candidates.json` 中补齐“拒绝原因”和中间评分
- 分离 occupancy / axis / facing / species / hazard 各自置信度
- 审查模式从 cattle 扩展到所有高风险识别类型
- 修正边缘 cattle 的过严过滤
- 改进 wolf / 装饰图案 / 顶部特效的误检抑制
- 增加多帧时序一致性，识别结果参考前一帧状态
- 做完整的手工棋盘编辑器，允许人工修正后保存样本
- 把全盘健康度做成统一评分和退出码

### 求解与执行

- 区分 `complete_solution`、`partial_hint`、`proven_unsat`、`timeout`、`node_cap`
- partial cache 不应阻止更高预算或新算法再次尝试
- 完整缓存回放前要验证 anchor、剩余数和起始盘面
- 加入 SCC 死锁、永久障碍阻塞等结构性剪枝
- 研究强制 EXIT 宏步规约
- 统一 CLI 与 GUI 的算法选择策略
- 增加动态 hazard 的在线模型
- 提供更清晰的执行模式：sandbox / single-step / auto / unarmed
- 使用更稳的输入发送方式，并恢复鼠标原位

### 缓存与数据

- `_solution.png` 不应在检测阶段复制，应在求解成功后按匹配哈希生成
- 每次运行先写 session staging，验证通过再原子发布
- 为缓存文件增加 schema/version/provenance/hash
- 拆分 level id、state hash、observation hash
- `learning_summary.json` 目前只写不读，需要落地成真正可用的学习反馈
- 给 mismatch 分类，而不是只统计 match / mismatch
- 增加缓存浏览、导出、去重、清理、保留策略
- 长期可考虑 SQLite 或可重建索引，降低 JSONL 零散文件成本

## P2：工程与产品体验

- 热键、窗口标题、点击节奏等都应进入 UI 配置，而不是硬编码
- 限制单实例运行，避免热键和缓存互相打架
- 补依赖清单、锁文件、`.gitignore`、CI
- 把超大脚本继续拆模块，减少 `detect_occupancy.py` / 前端状态逻辑的集中复杂度
- CLI 补齐明确退出码
- 解法图支持逐步动画或 step-by-step 播放
- 日志轮转和上下文信息更完整
- sandbox 页增加搜索、筛选、缩放、只看异常样本
- README 需要同步真实状态，当前仍写着点击执行未实现

## 测试缺口

- hazard 与 hazard-piece overlap
- `solved:false, remaining:0` 边界
- `axis_both` 在大盘搜索路径
- slide mode / species / facing 非法输入
- 缓存损坏、schema 升级、并发写入
- 使用现有 76 份快照做识别回归集
- WinAPI / DPI / 热键 / cancel mock
- 非棋盘场景、结算场景、弹窗场景绝不点击的端到端测试

## 建议的下次执行顺序

1. 先做“禁止误点”闭环：场景状态机、执行 blocker、单步执行、preflight、急停。
2. 再做“缓存可信”闭环：state hash、solution revision、board schema、hazard overlap 校验。
3. 然后修搜索边界 bug：`remaining:0`、`axis_both`、heuristic hazard 感知。
4. 最后再推进识别增强、缓存产品化、UI/工程体验。

## 备注

- 当前缓存中已有真实反馈样本，可直接作为下一轮回归测试基线。
- 文档刻意优先收录“高风险错误执行”和“错误缓存复用”相关项，因为它们最影响这类自动点击工具的可信度。
